# 11. 開発順序・開発者としての視点

## 開発順序（ポートフォリオ最大化）

「すべてのルールを先に決めよう」とする必要はない。真に難しいのは**シフト最適化ロジック**（難易度の 7 割）であり、OR-Tools モデルに**半日兼務・最低人数・希望休・勤務可能曜日・公平性**を満たす形で組み込むことが、ちょっとした数理パズルになる。

**詰まりにくくするための推奨順序:**

1. **DB 設計確定**
2. **認証・マルチテナント**
3. **OR-Tools モデル設計**
4. **API**
5. **フロント**

| Phase | 内容 |
|-------|------|
| **Phase 1: Core Foundation** | Docker 環境構築, Supabase 接続, DB スキーマ作成, JWT 認証, RBAC |
| **Phase 2: Multi-Tenant Security** | org 作成 API, 招待機能（invite + signup）, subscription 制御, max_users 制御 |
| **Phase 3: Business Logic** | 職員マスター管理, 希望休登録, シフト生成（OR-Tools: 変数・制約・目的関数）, 解なし処理 |
| **Phase 4: Production Hardening** | RLS 有効化, audit_logs 導入, エラーログ強化, Railway デプロイ, README 整備 |
| **Phase 5: Frontend** | org_admin 画面 3（職員管理・シフト生成・シフトカレンダー）, staff 画面 2（自分のシフト・希望休） |

## 開発者としての視点（要件の扱い）

ソフトウェアの要件は生き物だ。実装すると必ず変わる。これは避けられない自然現象に近い。

重要なのは**「完璧な仕様を書くこと」ではなく「変更に耐える構造」を作ること**である。

Shiftora の場合、その核心は次の 3 つになる。

- **データモデル**
- **最適化アルゴリズム**
- **マルチテナント構造**

この 3 つが綺麗なら、残りはあとからどうにでもなる。

そして、このプロジェクトで一番頭を使う部分はまだ手をつけていない。**OR-Tools のシフト最適化モデル設計**である。ここはちょっとしたパズルだ。変数・制約・目的関数をどう組み立てるか。うまく作ると、機械が「人間より公平なシフト」を作り始める。ソフトウェアの中でもなかなか楽しい種類の魔法である。
