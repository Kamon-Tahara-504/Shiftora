# 03. ロール・権限・ユースケース

## システムロール（プラットフォーム側）

| ロール | 権限 |
|--------|------|
| `super_admin` | org 作成 / org_admin 作成 / 契約変更 |
| `support_admin` | 閲覧・サポート |
| `billing_admin` | 契約管理 |

---

## 組織ロール

| ロール | 権限 |
|--------|------|
| `org_admin` | staff 招待 / 職員管理 / シフト生成 |
| `staff` | 自分のシフト閲覧 / 希望休提出 |

**1 組織の org_admin は複数可能**とする。現場では管理者が複数いるケースが普通にある。

API では RBAC により、上記ロールに応じて `/org/*`（org_admin）、`/staff/*`（staff）、`/admin/*`（system_role）のアクセスを制御する。

---

## 典型的ユースケース（一本道）

このストーリーが通ればシステムは成立する。

1. 管理者が組織を作成する
2. 管理者が職員を登録する
3. 職員に招待リンクを送る
4. 職員がログインして希望休を登録する
5. 月末に管理者がシフト生成を実行する
6. アルゴリズムがシフト案を生成する
7. 管理者が不足枠を確認する（解なし時）
8. 必要に応じて手動修正する
9. 職員が自分のシフトを確認する
