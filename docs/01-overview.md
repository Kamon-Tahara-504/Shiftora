# 01. プロジェクト概要

## 用語定義

システム設計では言葉の定義が世界のルールになる。Shiftora では次の 5 つを共通言語とする。

| 用語 | 意味 |
|------|------|
| **slot** | 勤務時間の単位（AM / PM） |
| **department** | デイサービス（daycare） or 訪問介護（visit） |
| **shift** | 職員が slot と department に割り当てられた 1 件 |
| **day_off** | 職員が希望する休み（希望休） |
| **cross_assignment** | 同日で部署をまたぐ勤務（半日兼務） |

---

## 目的

- 手作業シフト作成の削減
- 最低配置条件未達の防止
- 半日兼務の自動反映
- マルチテナント SaaS 設計の実装

---

## MVP 出荷条件（ゴールライン）

以下 5 つが動けば MVP 完成とする。

1. **org_admin が職員を登録できる**
2. **staff が希望休を登録できる**
3. **org_admin が 1 ヶ月のシフトを生成できる**
4. **生成されたシフトを保存できる**
5. **staff が自分のシフトを閲覧できる**

CSV・PDF 出力は後回しでよい。現場では便利だが、シフト最適化エンジンとは独立した機能だからである。

---

## 対象事業所スコープ（MVP）

スコープは狭く固定したほうが設計が強くなる。

- **単一事業所**
- **部署は 2 種類**（デイサービス / 訪問介護）
- **最低人数ルールは固定**（デイ 4 人以上、訪問 AM/PM 各 1 人、半日兼務 2 人）
- **複数拠点は未対応**

将来 multi-facility に拡張する場合でも、DB 設計は変えずに `facility_id` を追加するだけで済む。よくある進化パターンである。
